写给未来的自己

发表于 2017-10-04

最近几个月，我都没有在凌晨2点之前睡着了

最近几个月，我都没有在凌晨2点之前睡着了。最近总是有一些烦心事乱糟糟的让人心烦意乱， 今天睡了两次都被短暂的梦魇惊醒，之前折腾过一些博客，发现用户体验不好，UI设计很烂， 但是QQ空间之类的总是有太多不想让担心的人看到，所以很久没有写过博客了，我本身是很喜 欢把情绪记录下来的，最近急躁烦躁的很，身体也在作贱，感觉自己像是一个废物一样，虽然 很多人说不要这样妄自菲薄，我也希望自己快振作起来，但是发现对好多事情都没有那么有 兴趣，不知道未来是为了做什么，反而萌生出一种无欲无求的欲望，感觉很像是高中逃课的那种感觉。

我反复思考我之前做出的每个决定

我反复思考我之前做出的每个决定，造成现在的后果，以及让自己总是陷入不好的境地，第一件事情，就是反思，我反复思考我作出选择的原因，错误要付出代价。我并没有为之前的选择而十分懊恼，这中间的酸甜是我为自己选择叫的学费。之前出国留学最终因为一些原因而停止，马上投入工作中去又存在经验不够多以及深度不够深的尴尬境地，说实话，我本身不是怕什么，不够就承认就补充，就怕自己逃避贪图享受，从而让自己更加堕落而没有约束。

在我23-24岁的这没有工作的半年里，我偶尔感到无助

在我23-24岁的这没有工作的半年里，我偶尔感到无助。这种无助是环境中没有你期待的同类的孤独，愈发明白一个优秀的 人群带给你的动力和影响，看了一些书，对于之前对于某些问题的憧憬不能仅仅限于自己的想象，举个例子来说， 如果你想要变的有技术，应该多看公认的大牛在做什么，而不是自己觉得自己应该怎么搞，最后又不深入，这是 很不好的一件事情，术业有专攻，但是专攻中也需要更加深入的学习和聚集精力。

我的感情说实话我不太想说太多

我的感情说实话我不太想说太多，我觉得这不适合公众议论，毕竟是很戳心的事情。我本身不想伤害谁，但是也不想拖累谁， 有时候自己的决定自己都觉得可怕，但是不作出取舍总是会让双方更加煎熬，我期待美好的感情和厚脸皮的对方，就好像《半岛铁盒》 “已经习惯不去阻止你,过好一阵子你就会回来”那种感情，也好像《心雨》”好想告诉我的她…这里像副画” 那种处处挂念的美好。 我不是任性，也不是长不大，而是有时候真的怕，怕心痛的太厉害了。

未来的路

未来的路到底如何走？遇到每个需要的解决的问题，按格式化解决，同时区分工作和事业。加强金融方面的知识储备，考取一些证书， 为未来储备更好的金融知识，我相信技术在变，世界每天都在变，核心思想不变，一个人想技能做好，就要打好基础。举个例子， 一个人想富有，就要打好金融知识基础，知道金钱流通的基本原理，虽然之前在狗狗币中一天就能盈利几千块，但是风险还是不可控。

现在的打算

现在的打算是在10.1之后面一下老东家，再面一些中小公司，争取把工作定了，挣点钱，不能那么飘，让别人觉得不可靠，这样对个人 的认可度和本身的信心也是一种打击，一定要把自己从这单纯的几个月抽出来，放到人生的长河中去看待问题，我相信未来可期，我也相信 做难事必有所得。

优化网页性能

发表于 2017-10-04

优化网页性能 — 加载性能

本篇文章主要讲解优化网络性能，包括两个大部分：1. 加载性能 2. 渲染性能

加载性能

不同网络条件下如何更友好的加载页面，从以下几方面考虑问题

A. 优化内容效率
1. 避免不必要的下载
    a.清点您的网页上的自有资产和第三方资产。
    b.评估每个资产的表现：其价值及其技术性能。
    c.确定这些资源是否提供了足够的价值。

2. 优化基于文本的资产的编码和传送大小
    a.源码压缩：预处理和环境特定优化
    b.通过 GZIP 压缩文本

3. 图像优化
    a. 选择最合适的图片和图片大小

4. 网页字体优化
    a. 通过 @font-face 定义字体系列
    b. 优化加载和渲染,否则导致渲染三秒问题

5. HTTP 缓存
    a. 通过 ETag 验证缓存的响应
    b. Cache-Control

6. 模拟网络工具
    1. 浏览器：Chrome DevTools
    2. MAC系统工具: Network Link Conditioner
    3. 手机设备模拟：Android Emulator
    4. 使用不同位置和网络测试: WebPagetest
    5. 模拟调制解调器的速度: Fiddler
    6. HTTP/HTTPS 代理: Charles
B. 不加载两次相同的内容
1. web储存
2. 离线的UX交互
C. 注意加载顺序
1. 关键渲染路径优化
D. PRPL模式
PRPL 是一种用于结构化和提供 Progressive Web App (PWA) 的模式，该模式强调应用交付和启动的性能。
特性：
    1. 推送 - 为初始网址路由推送关键资源。
    2. 渲染 - 渲染初始路由。
    3. 预缓存 - 预缓存剩余路由。
    4. 延迟加载 - 延迟加载并按需创建剩余路由
应用结构：
    1.应用的主_进入点_从每个有效的路由提供。 此文件应非常小，它从不同网址提供，因此会被缓存多次。 进入点的所有资源网址都需要是绝对网址，因为它可以从非顶级网址提供。
    2.Shell 或 App Shell，包含顶级应用逻辑、路由器，等等。
    3.延迟加载的应用_片段_。片段可以表示特定视图的代码，或可延迟加载的其他代码（例如，首次绘制不需要的部分主应用，如用户与应用交互前未显示的菜单）。Shell 负责在需要时动态导入片段
应用执行：
    1.应用进入点
        进入点必须导入和实例化 Shell，并且有条件地加载任何所需的 polyfill。
        进入点的主要注意事项为：
          a.具有最少的静态依赖项，换言之，除了 App Shell 本身外并没有多少依赖项。
          b.有条件地加载所需的 polyfill。
          c.为所有依赖项使用绝对路径。
    2.App Shell
        Shell 负责路由，通常包含应用的主导航 UI。
        应用会根据需要延迟加载片段。例如，当用户更换到新路由时，应用会导入与该路由相关的片段。这可能会向服务器发出新请求，或者只是从缓存加载资源。
        Shell（包括其静态依赖项）应包含第一次绘制所需的全部资源。
    3.使用服务器推送
        a.设计用于服务器/浏览器组合的未捆绑构建，此类组合支持 HTTP/2，以便在优化缓存时传输第一次快速绘制所需的资源。可以使用 <link rel="preload">或 HTTP/2 Push 高效地触发这些资源的传输。
        b.对于不处理 HTTP/2 的浏览器，构建过程会生成一组不同的捆绑包：一个用于 Shell 的捆绑包，以及一个用于每个片段的捆绑包。
