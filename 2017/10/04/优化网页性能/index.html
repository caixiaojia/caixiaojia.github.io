优化网页性能

发表于 2017-10-04

优化网页性能 — 加载性能

本篇文章主要讲解优化网络性能，包括两个大部分：1. 加载性能 2. 渲染性能

加载性能

不同网络条件下如何更友好的加载页面，从以下几方面考虑问题

A. 优化内容效率
1. 避免不必要的下载
    a.清点您的网页上的自有资产和第三方资产。
    b.评估每个资产的表现：其价值及其技术性能。
    c.确定这些资源是否提供了足够的价值。

2. 优化基于文本的资产的编码和传送大小
    a.源码压缩：预处理和环境特定优化
    b.通过 GZIP 压缩文本

3. 图像优化
    a. 选择最合适的图片和图片大小

4. 网页字体优化
    a. 通过 @font-face 定义字体系列
    b. 优化加载和渲染,否则导致渲染三秒问题

5. HTTP 缓存
    a. 通过 ETag 验证缓存的响应
    b. Cache-Control

6. 模拟网络工具
    1. 浏览器：Chrome DevTools
    2. MAC系统工具: Network Link Conditioner
    3. 手机设备模拟：Android Emulator
    4. 使用不同位置和网络测试: WebPagetest
    5. 模拟调制解调器的速度: Fiddler
    6. HTTP/HTTPS 代理: Charles
B. 不加载两次相同的内容
1. web储存
2. 离线的UX交互
C. 注意加载顺序
1. 关键渲染路径优化
D. PRPL模式
PRPL 是一种用于结构化和提供 Progressive Web App (PWA) 的模式，该模式强调应用交付和启动的性能。
特性：
    1. 推送 - 为初始网址路由推送关键资源。
    2. 渲染 - 渲染初始路由。
    3. 预缓存 - 预缓存剩余路由。
    4. 延迟加载 - 延迟加载并按需创建剩余路由
应用结构：
    1.应用的主_进入点_从每个有效的路由提供。 此文件应非常小，它从不同网址提供，因此会被缓存多次。 进入点的所有资源网址都需要是绝对网址，因为它可以从非顶级网址提供。
    2.Shell 或 App Shell，包含顶级应用逻辑、路由器，等等。
    3.延迟加载的应用_片段_。片段可以表示特定视图的代码，或可延迟加载的其他代码（例如，首次绘制不需要的部分主应用，如用户与应用交互前未显示的菜单）。Shell 负责在需要时动态导入片段
应用执行：
    1.应用进入点
        进入点必须导入和实例化 Shell，并且有条件地加载任何所需的 polyfill。
        进入点的主要注意事项为：
          a.具有最少的静态依赖项，换言之，除了 App Shell 本身外并没有多少依赖项。
          b.有条件地加载所需的 polyfill。
          c.为所有依赖项使用绝对路径。
    2.App Shell
        Shell 负责路由，通常包含应用的主导航 UI。
        应用会根据需要延迟加载片段。例如，当用户更换到新路由时，应用会导入与该路由相关的片段。这可能会向服务器发出新请求，或者只是从缓存加载资源。
        Shell（包括其静态依赖项）应包含第一次绘制所需的全部资源。
    3.使用服务器推送
        a.设计用于服务器/浏览器组合的未捆绑构建，此类组合支持 HTTP/2，以便在优化缓存时传输第一次快速绘制所需的资源。可以使用 <link rel="preload">或 HTTP/2 Push 高效地触发这些资源的传输。
        b.对于不处理 HTTP/2 的浏览器，构建过程会生成一组不同的捆绑包：一个用于 Shell 的捆绑包，以及一个用于每个片段的捆绑包。
